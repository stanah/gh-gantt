# gh-gantt: なぜこのツールを作るのか

## 解決する課題

### 1. プロジェクト全体の構造が見えない

GitHub Projects (V2) はタスクをボードやテーブルで整理できる。ロードマップビューでタイムラインも表示できる。しかし以下が欠けている：

- **タスクの親子関係**（エピック → フィーチャー → タスク）の可視化
- **依存関係**（何が何をブロックしているか）の可視化
- **階層的な進捗の集約**（子タスクの完了が親に反映される）

個々のタスクの状態は見えても、「プロジェクトは予定通り進んでいるか」「ボトルネックはどこか」「スケジュールは間に合うか」が判断できない。

gh-gantt はこれらをガントチャートで統合的に可視化する。

### 2. AI エージェントがプロジェクト管理を操作できない

GitHub Projects の操作は Web UI に閉じている。GraphQL API はあるが、認証・複雑なクエリ・レートリミットなど、AI エージェントが気軽に使えるものではない。

gh-gantt は CLI インターフェースを提供する。CLI であれば：

- Claude Code のような AI エージェントが直接操作できる
- 人間と AI が同じコマンド・同じワークフローでタスクを管理できる
- スクリプトやフックによる自動化も可能

### 3. AI エージェントのセッション間でコンテキストが失われる

AI エージェントは会話が切り替わるとコンテキストを失う。前の会話で何をしていたか、プロジェクトがどういう状態かを覚えていない。

gh-gantt はプロジェクトデータをローカルファイル（`.gantt-sync/`）に保持する。新しい会話でも `gh-gantt status` や `gh-gantt list` を実行するだけで、プロジェクトの現在地を即座に把握できる。API コールもネットワーク接続も不要。

## 設計原則

これらの課題から、以下の設計原則が導かれる：

- **CLI ファースト** — すべての操作は CLI で完結する。Web UI は可視化の手段であり、操作の主体ではない
- **ローカルファースト** — プロジェクトデータはローカルに保持する。API は同期時のみ使用する
- **git ライクな同期モデル** — pull / push / conflict resolve で開発者が馴染みのあるメンタルモデルを提供する
- **GitHub Projects との共存** — 既存の GitHub ワークフローを置き換えるのではなく、足りない部分を補完する
