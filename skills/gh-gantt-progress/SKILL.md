# gh-gantt Progress

プロジェクトの進捗評価とタスク状態の管理を行う。

## 分析項目

| 項目 | 内容 |
|------|------|
| エピック進捗率 | 子タスクの完了数 / 全体数 |
| 遅延・閉じ忘れ | 実装済みだが open のままのタスクを検出。git log と完了条件の突合で判定 |
| リスク評価 | 期限が近いが未着手のタスク |
| ブロッカー停滞 | blocked_by が open のまま放置されているタスク |
| タスク衛生 | 親なし、日程なし、body 空、型不適切（task に子タスクがある等） |
| 次タスク提案 | 期限・依存関係・優先度を考慮して着手すべきタスクを提案 |

## プロセス

### Step 1: 同期

**REQUIRED:** `gh-gantt-sync`（pull）を invoke して最新データを取得する。

### Step 2: 状況の表示

以下を実行し、結果をそのまま表示する:
```bash
gh-gantt task list --state open --sort end_date
```

### Step 3: 分析範囲の確認

一覧を表示した上で、ユーザーにどの分析を行うか確認する。選択肢と該当件数を提示すること:

- **遅延・閉じ忘れ** — git log から最近の変更を確認し、関連タスクがクローズ可能か判定する
- **エピック進捗** — エピックごとの子タスク完了率を確認する
- **リスク評価** — 期限が近いが未着手のタスクを検出する
- **ブロッカー停滞** — blocked_by が open のまま放置されているタスクを検出する
- **タスク衛生** — 親なし・日程なし・body 空・型不適切を検出する
- **次タスク提案** — 着手すべきタスクを提案する
- **全分析** — 上記すべてを実行する

ユーザーの指示から分析の種類が明確な場合（例:「閉じ忘れはある？」→ 遅延・閉じ忘れ）、種類の確認は省略してよい。
ただし **遅延・閉じ忘れ** の場合は git log の対象範囲を確認する（Step 4 参照）。

### Step 4: 遅延・閉じ忘れの検出

#### 4a: git log の確認

まずユーザーに git log の対象範囲を確認する:
- 直近 N コミット / N 日間
- 特定のタグやリリース以降
- 全履歴

```bash
git log --oneline --since="2 weeks ago"   # 例: 直近2週間
git log --oneline v0.1.0..HEAD            # 例: タグ以降
```

#### 4b: コミットからタスクへのマッピング

git log の出力から関連する Issue 番号を特定する:
- コミットメッセージ内の `#N`、`Closes #N`、`Fixes #N` を探す
- Issue 番号が見つかったタスクについて `gh-gantt task show <id>` で状態を確認する
- open のタスクが閉じ忘れ候補

#### 4c: コミットメッセージに Issue 番号がない場合

コミットメッセージだけでは特定できない変更については、エージェンティックに調査する:
- `git diff` や `git show` で変更内容を確認する
- 変更されたファイル名・関数名・コンポーネント名をキーワードにして、オープンタスクの body や title を `gh-gantt task list --search <query>` で検索する
- Grep / Glob / Read でコードベースを検索し、タスクの完了条件との関連を確認する
- LSP が利用可能であれば、定義ジャンプや参照検索で実装の有無を確認する

### Step 5: 完了判定

Step 4 で特定した閉じ忘れ候補について `gh-gantt task show <id>` で body を読み、判定する。

**通常タスク（task, feature, bug）:**
1. body の完了条件を一つずつリストアップする
2. 各条件について、git diff の変更内容やコードベースの検索（Grep, Glob, Read, LSP 等）で実装されているか確認する
3. 全条件を満たしていれば「クローズ可」、一部でも未実装なら「未完了」と判定し、未実装の条件を明記する
4. body が空の場合はタイトルから判断し、「body が空のためタイトルで判断」と明記する

**エピック:**
1. 子タスクの状態を確認する（全 closed かどうか）
2. エピック自体の body も確認し、子タスクでカバーされていない要件がないか確認する

### Step 6: 結果提示

分析結果を一覧で提示し、アクション（クローズ、日程調整、ブロッカー解消等）を提案する。
